- 填写日期：2026-04-23

## 第1章 作品概述

### 1.1 创意来源与产生背景

极端高温正在从“天气现象”转变为“公共健康与基层治理问题”。对城市一般居民而言，高温意味着出行不适；但对高龄老人、慢病人群和行动能力较弱群体而言，高温可能直接转化为热暴露风险、就医风险和居家安全风险。现有很多高温预警只告诉公众“今天很热”，却没有继续回答三个更关键的问题：

1. 哪些空间区域里的老年群体最脆弱。
2. 这些高风险人群能否在短时间内到达避险资源。
3. 如果新增资源有限，最应该把资源投向哪里。

本作品正是在这一现实背景下提出。我们将项目落点放在武汉主城区，融合真实公开天气、老年人口、步行路网、公共服务设施和官方纳凉点通报数据，构建一套适用于街道、社区、民政、卫健和应急协同使用的高温避险辅助决策系统。

### 1.2 用户群体与应用对象

本作品不是面向普通消费者的资讯网页，而是面向明确治理主体的辅助决策工具，目标用户包括：

- 街道办与社区治理人员：用于高温日常巡查、重点片区排查和临时纳凉点布设。
- 民政与卫健部门：用于识别高龄暴露集中区和服务盲区。
- 应急管理部门：用于热浪来临前的资源预布设和联动调度。
- 学校、科研与展示场景：用于演示真实数据如何转化为城市治理决策。

### 1.3 主要功能与作品特色

本作品当前版本已经形成从数据更新到系统展示的完整链路，主要包括以下功能：

1. 自动接入多源真实公开数据，包括 Open-Meteo、WorldPop、OpenStreetMap/Overpass、Geofabrik 和武汉官方纳凉通报。
2. 构建网格级老年热健康风险图层，识别重点街区与高风险暴露区。
3. 基于真实步行路网计算可达性，评估现有资源在 5/10/15 分钟阈值下的覆盖能力。
4. 基于有限新增点位约束，输出临时纳凉点 `3/5/8` 点布设方案。
5. 对风险模型、路网分析和选址优化进行量化实验验证。
6. 增加“数据真实性审计”模块，明确区分真实上游输入、模型推导输出和代理变量。

### 1.4 应用价值与推广前景

本作品的核心价值，不是做出一张“哪里热”的图，而是把高温治理从粗颗粒提醒推进到可解释、可量化、可执行的资源调度层面。它具有以下应用价值：

- 对基层治理：帮助街道和社区找到最需要优先照顾的人群与区域。
- 对公共服务：帮助有限资源更精准地投向高风险服务盲区。
- 对社会治理：提升老龄化背景下的城市气候适应能力。
- 对推广复制：只要替换研究区域和输入数据，即可迁移到其他城市。

## 第2章 问题分析

### 2.1 问题来源

高温预警是常见公共服务，但在基层场景中，真正难的不是“知道会热”，而是“知道热浪来了以后先做什么”。传统天气预警通常按城市级或行政区级发布，难以进一步定位到街区、网格甚至步行服务范围。另一方面，老年群体对高温更敏感，且往往更依赖步行可达的社区资源、绿地空间、社区中心和应急服务点。这意味着高温治理不能只看气象值本身，还必须和人口结构、空间环境、服务资源和到达成本一起分析。

### 2.2 现有解决方案

现有相关方案大致分为三类：

#### 2.2.1 单一天气预警类

这类方案以温度、体感温度或热指数为主要依据，优点是实现简单、更新频率高，但通常存在以下不足：

- 颗粒度粗，只能告诉用户“一个区很热”，不能细化到服务盲区。
- 没有与老年人口暴露强度联动。
- 不回答资源是否够、资源该怎么调。

#### 2.2.2 静态资源盘点类

部分系统会展示公园、医院、社区服务中心或纳凉点清单，但问题在于：

- 只是“有哪里”，不是“能否及时到达”。
- 很少基于真实步行路网计算。
- 没有把资源供给与高风险人群需求量化关联。

#### 2.2.3 学术研究或单环节算法类

已有研究会分别关注高温脆弱性识别、城市公园可达性、冷却中心选址优化等方向，但很多工作停留在单环节：

- 只做风险图，不做路网诊断；
- 只做可达性，不做资源优化；
- 只做算法，不做可运行系统；
- 很少明确说明数据真实性边界与代理变量使用范围。

### 2.3 本作品要解决的痛点问题

基于上述分析，本作品要解决的不是单一预测问题，而是四个连续痛点：

1. 如何识别真正需要优先干预的高龄热暴露区。
2. 如何判断这些区域当前是否被已有资源有效覆盖。
3. 如何在新增点位有限时得到收益更高的布设方案。
4. 如何在比赛材料中把“真实数据”和“模型输出”的边界讲清楚，避免夸大。

### 2.4 解决问题的思路

本作品采用“真实数据输入 + 多层模型推导 + 系统化展示”的总体思路：

1. 使用真实公开天气、人口、路网、POI 和官方通报作为上游输入。
2. 将研究区划分为网格，构建老年热健康风险识别模型。
3. 基于真实步行路网计算到现有资源的到达时间与覆盖率。
4. 以高风险老年人口为需求端、以公园和图书馆等公共设施为候选端，建立选址优化模型。
5. 通过风险验证、路网对比、情景实验和消融实验验证系统有效性。
6. 通过真实性审计模块，明确说明哪些结果可以被称为“真实公开数据输入”，哪些只能被称为“模型推导输出”。

本章提出的指标在第5章得到量化印证。

### 3.2 数据来源与预处理方案

#### 3.2.1 天气数据

- 来源：Open-Meteo Forecast API 与 Open-Meteo Archive API。
- 当前系统默认风险场景：`historical_heatwave_case`。
- 当前历史热浪窗口：`2025-08-03T07:00` 至 `2025-08-06T06:00`。

之所以采用该窗口，是因为当前时间处于 2026 年 4 月，实时气象并未形成显著热浪。为了让系统在比赛演示时仍能展示真实高温治理场景，项目采用真实历史热浪窗口进行风险推演，而不是人为抬高温度。

#### 3.2.2 老年人口数据

- 来源：WorldPop 中国 2025 年 1km 老年人口栅格。
- 当前版本：`R2025A`。
- 当前数据级别：`worldpop_raster`。

该数据不是街道逐户登记原始表，但属于真实公开人口栅格输入，可作为研究区老年人口空间分布的重要依据。

#### 3.2.3 POI 与官方纳凉点数据

- 公共设施来源：OpenStreetMap / Overpass API。
- 官方纳凉点来源：武汉市政府、民政局、国动办等公开页面。

系统当前监测官方页面 `7` 个，全市通报纳凉点总量 `1763` 个，已完成原文和空间核验、可进入研究区分析的官方在运点位 `5` 个。

#### 3.2.4 路网数据

- 来源：Geofabrik 湖北 OSM 路网数据。
- 当前图规模：`174973` 个节点、`195564` 条边。
- 当前正式数据级别：`walk_network`。

该部分是作品技术可信度的重要来源，因为正式结论优先基于真实步行路网，而非欧式直线距离。

### 3.3 老年热健康风险识别模型

本作品的风险模型并不是单纯看温度，而是综合考虑以下因素：

- 气象热负荷：温度、体感温度、持续高温窗口；
- 老年人口暴露：65 岁及以上人口空间集中度；
- 局地环境代理：建筑覆盖、道路密度、绿地和近公园距离等；
- 服务可达性修正：步行到避险资源的时间成本。

因此，风险网格的含义不是“该格子的原始实测温度”，而是“在真实高温情景下，该格子对高龄群体的综合脆弱性强弱”。这一点在本项目的数据真实性审计中被明确标注为“模型推导输出”。

### 3.4 步行可达性分析模型

可达性分析以真实步行路网为主，针对研究区内的现有资源计算：

- 平均最近步行时间；
- 5 分钟覆盖率；
- 10 分钟覆盖率；
- 15 分钟覆盖率；
- 高风险网格的可覆盖情况。

为了验证真实路网是否必要，项目还同时保留了距离代理方法作为对照组，在第5章给出误差分析。

### 3.5 临时纳凉点选址优化模型

在优化问题中：

- 需求端：高风险老年人口网格；
- 基线资源：既有主动避暑资源；
- 候选点：公园与图书馆等可转化公共设施；
- 约束：新增点位数量、15 分钟阈值、服务能力与适配度；
- 目标：优先最大化高风险人群覆盖，再压缩平均到达时间。

当前策略为：`mclp_capacity_readiness_fairness_hybrid`，即在最大覆盖基础上叠加容量代理、开放时段代理、开放性代理、运行适配度与片区公平性修正。

需要说明的是，这里的容量、开放时段和适配度属于可解释代理变量，不是逐点实测运营数据。

### 3.6 数据真实性审计机制

为了避免把模型结果错误描述成“全部真实数据”，本作品新增真实性审计模块，对天气、人口、可达性、官方纳凉点、风险模型和选址优化六个模块逐项审计。

当前审计结论为：

- 总体结论：`真实公开数据输入 + 模型推导输出`
- `can_claim_all_data_real = false`
- `can_claim_all_upstream_sources_real = true`
- 模块数：`6`
- 含代理变量模块数：`2`
- 回退模块数：`0`

该机制保证比赛文档和系统展示口径统一，不夸大、不虚构。

## 第5章 测试分析

### 5.1 测试数据与实验设置

当前研究区风险网格总数为 `140`，其中高风险网格 `27` 个。系统同时使用：

- 历史热浪窗口作为默认推演场景；
- WorldPop 老年人口栅格；
- Geofabrik 真实步行路网；
- 官方纳凉点核验数据；
- 公园与图书馆候选点集。

实验重点围绕“模型是否更有效”“真实路网是否必要”“新增点位是否显著改善覆盖”三个问题展开。

### 5.2 风险模型验证

在相同 Top-27 网格规模下，各版本模型对老年人口的识别能力如下：

- 仅温度阈值：捕获老年人口 `102772`
- 温度 + 体感温度：捕获老年人口 `102772`
- 温度 + 体感温度 + 老年人口：捕获老年人口 `190551`
- 完整模型（加入可达性）：捕获老年人口 `213425`

这说明如果只看天气强度，会错过大量真正需要优先干预的高龄暴露区；而将人口结构和可达性纳入后，模型更接近实际治理需求。

### 5.3 可达性算法对比

真实步行路网与距离代理的结果对比如下：

- 真实路网平均最近步行时间：`15.09` 分钟
- 距离代理平均最近步行时间：`17.19` 分钟
- 平均绝对误差：`6.4` 分钟
- RMSE：`8.07` 分钟
- 过度乐观误判网格数：`34`

同时，在 15 分钟阈值下：

- 真实路网覆盖率：`31.43%`
- 距离代理覆盖率：`55.71%`

该结果表明，如果仅使用距离代理，会系统性高估资源可达性，掩盖真实服务短板。

### 5.4 选址优化情景实验

#### 5.4.1 基线结果

仅考虑既有主动避暑资源时：

- 覆盖高风险老年人口：`15617`
- 覆盖率：`7.32%`
- 平均到达时间：`35.75` 分钟

#### 5.4.2 新增 5 点方案

- 覆盖率提升到：`32.00%`
- 新增覆盖高风险老年人口：`52689`
- 平均到达时间下降到：`31.15` 分钟

#### 5.4.3 新增 8 点方案

- 覆盖率提升到：`41.18%`
- 新增覆盖高风险老年人口：`72280`
- 平均到达时间下降到：`27.82` 分钟

这表明在资源有限前提下，通过优化模型选择少量新增点位，就能显著改善服务覆盖。

### 5.5 消融实验

项目进一步进行模块消融，验证各模块是否真正有效：

1. 移除老年人口暴露后，TopN 网格老年人口捕获量从 `213425` 降至 `102772`，差值 `110653`。
2. 移除真实路网后，15 分钟覆盖率从 `31.43%` 被高估到 `55.71%`。
3. 移除选址优化后，高风险人口覆盖率仅维持在 `7.32%`，无法达到 `32.00%` 的改进水平。
4. 移除选址优化后，平均到达时间无法从 `35.75` 分钟下降到 `31.15` 分钟。

这些结果说明，本作品不是简单堆叠图表，而是每个模块都对最终效果有实质贡献。

### 5.6 数据真实性测试结论

项目额外回答评审最容易质疑的问题：“这些数据到底是不是都是真的？”

当前答案是：

- 上游数据是真实公开数据；
- 风险网格和选址推荐是基于真实数据计算出的模型输出；
- 其中局地热环境、容量、开放时段和运营适配度包含可解释代理变量；
- 当前未触发任何演示级数据回退。

这一点在系统前端和报告文档中已保持一致。

## 第6章 作品总结

### 6.1 作品特色与创新点

本作品的特色与创新主要体现在五个方面：

1. 选题切口真实。将高温治理与老龄化公共服务结合，问题明确，公共价值强。
2. 数据链路真实。接入真实天气、真实人口栅格、真实路网、真实官方通报。
3. 决策逻辑闭环。不是停留在风险展示，而是继续做可达性诊断和点位优化。
4. 实验体系完整。包含模型验证、路网对比、情景实验和消融实验。
5. 真实性表达严谨。通过审计模块明确说明哪些是真实输入、哪些是模型输出。

### 6.2 应用推广

本项目具有较强推广价值：

- 对其他城市：只需替换研究区和上游数据，即可迁移。
- 对其他灾害：方法可扩展到寒潮、暴雨后避险服务、脆弱人群保障等场景。
- 对基层治理：可作为街道和社区在极端天气期间的辅助排班、布点和巡查工具。
- 对教学科研：可作为“真实数据驱动的城市治理系统”示范案例。

### 6.3 作品展望

后续还可从四个方向继续增强：

1. 接入更细粒度树荫、楼栋、独居老人等数据；
2. 用真实场馆运营时段和容量替代代理变量；
3. 扩展更多类型候选设施，如社区驿站、文化站、养老服务站；
4. 补充多城市复制实验，形成更强的泛化能力证明。
